## 一、背景与动机

### 1.1 现状问题诊断

当前`src/lib/marriage-deep-report.ts`中的深度报告存在三类问题：

#### 🔴 内容重复（3组重叠）

| 重叠组 | 涉及章节 | 问题描述 |
|--------|---------|---------|
| 日支互动重复 | 第2节「日柱互动·合冲刑害」↔ 第3节「婚姻宫详解」 | 两节均分析日支与其他柱位的合冲刑害，第3节只是换了一种叙述方式 |
| 配偶星重复 | 第4节「十神格局」↔ 第5节「配偶星全维分析」 | 配偶星得令、比劫夺财/争官在两节中分别展开 |
| 用神重复 | 第8节「用神与姻缘」↔ 第10节「综合研判」 | 综合研判中重新打分和评用神，与第8节交叉 |

#### 🟡 不够专业

| 问题 | 涉及章节 |
|------|---------|
| 20+颗神煞全部展开，像词典不像报告，没有主次 | 第1节 |
| 五行感情特质纯模板（5种元素5套预设文本），与具体八字无关 | 第7节 |
| 14分加权评分体系在传统子平法中无出处，有"凑科学性"之嫌 | 第10节 |
| 十二长生12套预设文本，未关联其他维度 | 第6节 |

#### 🟠 内容深度不足

| 问题 | 说明 |
|------|------|
| 大运分析每运仅2-3句，非关键大运直接写"无直接引动" | 第9节 |
| 缺少流年逐年的个性化解读 | 缺失 |
| 大部分建议是通用的"多沟通多包容" | 贯穿全文 |
| 双人合盘仅5节，比免费结果（6维度）还单薄 | `buildPairDeepReport()` |

### 1.2 设计目标

1. **有依据**：每个论断追溯至具体的天干地支组合 + 经典规则 + 口诀出处
2. **不重复**：合并重叠章节，同一信息不在两处展开
3. **去模板**：所有文本基于实际八字数据计算生成，不用预设文案填充
4. **可追溯**：附录列明规则出处，用户可自行验证
5. **付费感**：双人版从5节扩展到8节，每节内容更扎实

---

## 二、架构设计：方案A+B混合

**方法**：采用传统子平法的章节框架（方案A）保证专业性和无重复，同时在每节标题和开头用问题化表述（方案B）让用户知道"这节回答什么问题"。关键结论附带推导依据（方案C精神）。

### 3.2 各节详细设计

#### 第1节：命盘总览

**数据结构**：直接使用`BaziResult`的全部字段。

**展示内容**：
- **四柱全表**：年/月/日/时 × 天干地支 / 藏干 / 纳音 / 十神 / 空亡
- **日主信息**：日主 + 五行 + 强弱（偏弱/中和/偏旺）+ 用神 + 生肖
- **命宫/身宫/胎元**：简要标注，不作为分析重点
- **白话导读**：2-3句日常语言解释盘面基本特征

**数据来源**：
```typescript

bazi.dayMaster     // 日主
bazi.dominantElement // 主导五行

bazi.zodiac        // 生肖

bazi.shenGong      // 身宫
bazi.taiYuan       // 胎元
```

#### 第2节：婚姻宫精析

**合并**：现「日柱互动·合冲刑害」+「婚姻宫详解」

**子模块**：
1. **婚姻宫基本信息**：日支是哪个地支、五行属性、全部藏干、纳音
2. **日主与婚姻宫的生克关系**：日主生婚姻宫 / 婚姻宫生身 / 比和 / 相克，各对应什么含义
3. **月柱冲合**（权重最高）：月支与日支的合冲刑害——外部环境（工作/家庭/社会）对婚姻的影响
4. **年柱/时柱冲合**：家庭背景/晚年的影响，权重和含义各自不同
5. **日干合相**：日干是否与月干/年干/时干五合
6. **藏干中的配偶信号**：日支藏干是否包含配偶星

**关键规则**：每个合冲刑害类型的分柱位含义已有良好基础（`analyzeDayBranchInteractions`），保留并继续使用。

**去重措施**：同一组合冲刑害不再在两节中分别叙述。

#### 第3节：配偶星全维分析

**合并**：现「十神格局·感情模式」+「配偶星全维分析」+「五行感情特质」（去模板化后融入十神）

**子模块**：
1. **配偶星定义**：男命财星 / 女命官杀（基于`gender`字段）
2. **出现情况**：天干透出次数 + 地支藏干次数 + 具体柱位
3. **质量评级**：保留现有四级体系（有力/中等/偏弱/隐伏）
4. **得令分析**：配偶星是否得月令生扶
5. **配偶画像**：基于配偶星五行和柱位的传统取象（带明确免责声明）
6. **十神格局信号**：官杀混杂 / 正偏财混杂 / 比劫夺 / 食伤克（仅在实际存在时列出）

## 五、技术实现要点

### 5.1 涉及文件

| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `src/lib/marriage-deep-report.ts` | 🔧 重构 | 主要变更文件，重写`buildSoloDeepReport`和`buildPairDeepReport` |
| `src/lib/marriage.ts` | 📈 扩展 | 可能需要新增辅助函数 |
| `src/lib/shensha.ts` | 📈 扩展 | 可能需要新增筛选/排序逻辑 |
| `src/lib/types.ts` | ❌ 不变 | 现有类型基本够用 |

### 5.2 数据流

```
用户输入 (MarriageFormData)
  ↓
calculateBazi() / calculateMarriage() / calculateSingleMarriage()
  ↓
免费结果 (MarriageResult / SingleMarriageResult)
  ↓ (付费后)
buildSoloDeepReport() / buildPairDeepReport()
  ↓
MarriageDeepReport { id, mode, headline, summary, baziOverview?, sections[] }
  ↓
/reports/marriage/[id] 页面渲染
```

### 5.3 类型变更

```typescript
// DeepReportSection 保持不变
export type DeepReportSection = {
  title: string;
  subtitle?: string;
  body: string;           // 主文本（Markdown）
  highlights: string[];   // 关键标签
  data?: Record<string, unknown>; // 结构化数据（表格/图表用）
};

// MarriageDeepReport 保持不变
// sections 数组从 10→8（单人）或 5→8（双人）
```

### 5.4 关键原则

1. **所有文本在服务端生成**：`buildSoloDeepReport` / `buildPairDeepReport` 是纯函数，输入八字数据，输出报告
2. **不引入AI生成**：所有文本基于规则推导，保证可复现、可验证
3. **数据决定内容**：同一套代码，不同八字产生不同的报告文本
4. **附录规则集中管理**：经典出处文本从常量中引用，不散落在各处
